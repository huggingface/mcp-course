# 模块 1：构建 MCP 服务器

## CodeCraft Studios 的 PR 乱象

这是您在 CodeCraft Studios 的第一周，您正在目睹一些让每位开发者都头疼的事情。团队的拉取请求 (Pull Request, PR) 看起来是这样的：

-   "stuff" (一些东西)
-   "more changes" (更多更改)
-   "fix" (修复)
-   "update things" (更新内容)

与此同时，代码审查的积压越来越多，因为审查者无法理解到底发生了什么变化，或者为什么要这样做。后端团队的 Sarah 花了 30 分钟试图弄清楚 "various improvements" (各种改进) 实际上意味着什么，而前端的 Mike 不得不翻阅 47 个文件来理解一个 "small fix" (小修复)。

团队知道他们需要更好的 PR 描述，但每个人都忙于交付功能，没有时间写详细的说明。他们需要一个既能提供帮助又不会拖慢进度的解决方案。

**您的任务**：构建一个智能的 PR 代理，它能自动分析代码变更并建议有用的描述。

## 概述

您的 PR 代理将使用 MCP 开发的一个关键原则来解决 CodeCraft Studios 的问题：不是硬编码关于什么是“好”的 PR 的僵化规则，而是向 Claude 提供原始的 git 数据，让它智能地建议适当的描述。

这种方法之所以有效，是因为：
-   **灵活的分析**：Claude 可以理解简单规则可能会遗漏的上下文。
-   **自然的语言**：建议感觉像是人类写的，而不是机器人。
-   **适应性强**：它适用于任何代码库或编码风格。

您将实现三个基本的工具，它们为整个自动化系统建立了模式：

1.  **analyze_file_changes** - 检索 git diff 信息和已更改的文件 (数据收集)。
2.  **get_pr_templates** - 列出可用的 PR 模板 (资源管理)。
3.  **suggest_template** - 允许 AI 推荐最合适的模板 (智能决策)。

### 设计理念

与那些基于文件扩展名或僵化模式对变更进行分类的传统系统不同，您的实现应该：

-   向 Claude 提供原始的 git 数据 (diffs、文件列表、统计信息)。
-   让 Claude 分析实际的代码变更。
-   允许 Claude 做出智能的模板建议。
-   保持逻辑简单——让 Claude 来处理复杂性。

> [!TIP]
> **MCP 理念**：不要在您的工具中构建复杂的逻辑，而是向 Claude 提供丰富的数据，让它的智能来做决策。这使您的代码比传统的基于规则的系统更简单、更灵活。

### 错误处理

始终优雅地处理潜在的错误：

```python
try:
    result = subprocess.run(["git", "diff"], capture_output=True, text=True)
    return json.dumps({"output": result.stdout})
except Exception as e:
    return json.dumps({"error": str(e)})
```

> [!WARNING]
> **错误处理**：即使是错误，也要始终从您的工具返回有效的 JSON。Claude 需要结构化的数据来理解出了什么问题，并向用户提供有用的响应。

### 处理大型输出 (关键学习时刻！)

> [!WARNING]
> **真实世界的约束**：MCP 工具每个响应有 25,000 个令牌的限制。大型的 git diff 很容易超过这个限制 10 倍或更多！这是生产环境 MCP 开发中的一个关键课题。

在实现 `analyze_file_changes` 时，您可能会遇到这个错误：
```
Error: MCP tool response (262521 tokens) exceeds maximum allowed tokens (25000)
```

**为什么会发生这种情况：**
-   单个文件的更改可能有数千行。
-   企业级仓库通常有大规模的重构。
-   `git diff` 默认包含完整的上下文。
-   JSON 编码增加了开销。

这教会了我们一个重要的原则：**在设计工具时，始终要考虑输出的限制**。以下是解决方案：

```python
@mcp.tool()
async def analyze_file_changes(base_branch: str = "main", 
                              include_diff: bool = True,
                              max_diff_lines: int = 500) -> str:
    """使用智能输出限制来分析文件变更。
    
    Args:
        base_branch: 要进行比较的分支。
        include_diff: 是否包含实际的 diff 内容。
        max_diff_lines: 要包含的最大 diff 行数 (默认为 500)。
    """
    try:
        # 获取 diff
        result = subprocess.run(
            ["git", "diff", f"{base_branch}...HEAD"],
            capture_output=True, 
            text=True
        )
        
        diff_output = result.stdout
        diff_lines = diff_output.split('\n')
        
        # 如果需要，进行智能截断
        if len(diff_lines) > max_diff_lines:
            truncated_diff = '\n'.join(diff_lines[:max_diff_lines])
            truncated_diff += f"\n\n... 输出已截断。共 {len(diff_lines)} 行，仅显示前 {max_diff_lines} 行 ..."
            diff_output = truncated_diff
        
        # 获取摘要统计信息
        stats_result = subprocess.run(
            ["git", "diff", "--stat", f"{base_branch}...HEAD"],
            capture_output=True,
            text=True
        )
        
        return json.dumps({
            "stats": stats_result.stdout,
            "total_lines": len(diff_lines),
            "diff": diff_output if include_diff else "使用 include_diff=true 来查看 diff",
            "files_changed": self._get_changed_files(base_branch)
        })
        
    except Exception as e:
        return json.dumps({"error": str(e)})
```

**处理大型输出的最佳实践：**
1.  **实现分页**：将大型结果分成多个页面。
2.  **添加过滤选项**：让用户可以请求特定的文件或目录。
3.  **首先提供摘要**：在返回完整内容之前，先返回统计信息。
4.  **使用渐进式披露**：从高级信息开始，允许用户深入了解细节。
5.  **设置合理的默认值**：默认为适用于大多数情况的合理限制。

## 工作目录注意事项

默认情况下，MCP 服务器在其安装目录中运行命令，而不是在 Claude 客户端的当前工作目录中。这意味着您的 git 命令可能会分析错误的仓库！

为了解决这个问题，MCP 提供了 [roots](https://modelcontextprotocol.io/docs/concepts/roots) 的概念——这是一种让客户端通知服务器相关目录的方式。像 Cursor 或 Claude Code 这样的客户端会自动提供其工作目录作为 root。

以下是如何在您的工具中访问它：

```python
@mcp.tool()
async def analyze_file_changes(...):
    # 从 roots 获取客户端的工作目录
    context = mcp.get_context()
    roots_result = await context.session.list_roots()
    
    # 从 FileUrl 对象中提取路径
    working_dir = roots_result.roots[0].uri.path
    
    # 用于所有 git 命令
    result = subprocess.run(
        ["git", "diff", "--name-status"],
        capture_output=True,
        text=True,
        cwd=working_dir  # 在客户端的目录中运行！
    )
```

这确保了您的工具在 AI 客户端实际工作的仓库上操作，而不是在 MCP 服务器的安装位置。

## 故障排查

-   **导入错误**：确保您已运行 `uv sync`。
-   **Git 错误**：确保您在一个 git 仓库中。
-   **无输出**：MCP 服务器通过 `stdio` 通信 - 使用支持的客户端 (如 Claude Desktop) 进行测试。
-   **JSON 错误**：所有工具都必须返回有效的 JSON 字符串。
-   **超过令牌限制**：这对于大型 diff 是预期的！请如上所示实现输出限制。
-   **"响应太大" 错误**：添加 `max_diff_lines` 参数或设置 `include_diff=false`。
-   **Git 命令在错误的目录中运行**：MCP 服务器默认在其安装目录中运行。要解决此问题，请使用 [MCP roots](https://modelcontextprotocol.io/docs/concepts/roots) 来访问客户端的当前目录：
    ```python
    # 从 roots 获取客户端的工作目录
    context = mcp.get_context()
    roots_result = await context.session.list_roots()
    working_dir = roots_result.roots[0].uri.path  # FileUrl 对象有一个 .path 属性
    
    # 在 subprocess 调用中使用它
    subprocess.run(["git", "diff"], cwd=working_dir)
    ```
    AI 客户端会自动提供其工作目录作为 root，允许您的 MCP 服务器在正确的位置进行操作。
