# 第三单元解决方案演练：使用 MCP 构建拉取请求智能体  (Pull Request Agent) 

## 概述

本演练将指导您完成第三单元中“PR Agent”的完整解决方案——这是一个 MCP 服务器，它通过分析代码变更、监控 CI/CD 流水线和自动化团队沟通，来帮助开发者创建更好的拉取请求。该解决方案展示了 MCP 的三种原语（工具、资源和提示）在实际工作流程中如何协同工作。

## 架构概述

这个 PR Agent 由多个相互连接的模块组成，它们逐步构建起一个完整的自动化系统：

1.  **构建 MCP 服务器** - 具备 PR 模板建议工具的基础服务器。
2.  **智能文件分析** - 使用“资源”提供项目上下文，从而进行增强分析。
3.  **GitHub Actions 集成** - 使用标准化的“提示词”进行 CI/CD 监控。
4.  **Hugging Face Hub 集成** - 为模型部署和数据集 PR 提供工作流。
5.  **Slack 通知** - 整合所有 MCP 原语，实现团队沟通。

## 模块 1：构建 MCP 服务器

### 我们正在构建什么
一个最小化的 MCP 服务器，它使用 MCP 工具来分析文件变更，并建议适当的 PR 模板。

### 关键组件

#### 1. 服务器初始化 (`server.py`)
```python
# 服务器注册了三个基本工具：
# - analyze_file_changes：返回关于已变更文件的结构化数据
# - get_pr_templates：列出可用的模板及其元数据
# - suggest_template：提供智能的模板推荐
```

服务器使用 MCP SDK 将这些工具暴露给 AI 模型 (如 Claude)，使其能够收集信息并就使用哪个 PR 模板做出智能决策。

#### 2. 文件分析工具
`analyze_file_changes` 工具会检查 git diff 以识别：
-   文件类型和扩展名
-   变更的文件数量
-   添加/删除的行数
-   常见的模式 (例如，测试、配置、文档的变更)

这些结构化的数据使 AI 能够理解变更的性质，而无需硬编码决策逻辑。

#### 3. 模板管理
模板以 Markdown 文件的形式存储在 `templates/` 目录中：
-   `bug.md` - 用于错误修复
-   `feature.md` - 用于新功能
-   `docs.md` - 用于文档更新
-   `refactor.md` - 用于代码重构

每个模板都包含占位符，Claude 可以根据分析结果进行填充。

### Claude 如何使用这些工具

1.  Claude 调用 `analyze_file_changes` 来了解发生了什么变更。
2.  使用 `get_pr_templates` 查看可用的选项。
3.  使用分析数据调用 `suggest_template`。
4.  接收带有推理过程的推荐。
5.  可以根据具体的变更来自定义模板。

### 学习成果
-   理解工具的注册和模式 (schema)。
-   让 Claude 使用结构化数据做出决策。
-   将数据收集与决策逻辑分离。

## 模块 2：智能文件分析

### 我们正在构建什么
使用 MCP 资源来提供项目上下文和团队指南，从而实现增强的文件分析。

### 关键组件

#### 1. 资源注册
服务器暴露了四种类型的资源：
```python
# 资源提供了对以下内容的只读访问：
# - file://templates/ - PR 模板文件
# - file://project-context/ - 编码标准、约定
# - git://recent-changes/ - 提交历史和模式
# - team://guidelines/ - 审查流程和标准
```

#### 2. 项目上下文资源
`project-context/` 目录包含：
-   `coding-standards.md` - 特定语言的约定
-   `review-guidelines.md` - 审查者关注的内容
-   `architecture.md` - 系统设计模式
-   `dependencies.md` - 第三方库策略

AI 可以读取这些内容以了解项目特定的要求。

#### 3. Git 历史分析
`git://recent-changes/` 资源提供：
-   最近的提交消息和模式
-   常见的 PR 标题和描述
-   团队成员的贡献模式
-   历史上的模板使用情况

这有助于 AI 建议与团队实践相一致的模板。

### AI 如何使用资源

1.  读取 `team://guidelines/review-process.md` 以了解 PR 的要求。
2.  访问 `file://project-context/coding-standards.md` 获取样式指南。
3.  分析 `git://recent-changes/` 以匹配团队的模式。
4.  将此上下文与文件分析相结合，以获得更好的建议。

### 增强的决策制定
有了资源，AI 现在可以：
-   建议符合团队约定的模板。
-   在 PR 中包含项目特定的要求。
-   在描述中引用编码标准。
-   与历史上的团队实践保持一致。

### 学习成果
-   资源 URI 的设计和模式。
-   使项目知识能被 AI 访问。
-   具有上下文感知能力的决策制定。
-   在自动化与团队标准之间取得平衡。

## 模块 3：GitHub Actions 集成

### 我们正在构建什么
使用 Webhook 和标准化的提示词进行实时 CI/CD 监控，以实现一致的团队沟通。

### 关键组件

#### 1. Webhook 服务器
使用 Cloudflare Tunnel 接收 GitHub Actions 事件：
```python
# Webhook 端点处理以下事件：
# - workflow_run 事件
# - check_run 事件
# - pull_request 状态更新
# - 部署通知
```

#### 2. 提示词模板
四个标准化的提示词确保了一致性：
-   **"分析 CI 结果"** - 处理测试失败和构建错误。
-   **"生成状态摘要"** - 创建人类可读的状态更新。
-   **"创建后续任务"** - 根据结果建议下一步。
-   **"起草团队通知"** - 为不同受众格式化更新。

#### 3. 事件处理流水线
1.  从 GitHub 接收 Webhook。
2.  解析事件数据并提取相关信息。
3.  根据事件类型使用适当的提示词。
4.  生成标准化的响应。
5.  存储以供团队通知。

### Claude 如何使用提示词

提示词使用示例：
```python
# 当测试失败时，AI 使用 "分析 CI 结果" 提示词：
prompt_data = {
    "event_type": "workflow_run",
    "status": "failure",
    "failed_jobs": ["unit-tests", "lint"],
    "error_logs": "...",
    "pr_context": {...}
}

# Claude 生成：
# - 根本原因分析
# - 建议的修复方案
# - 影响评估
# - 下一步行动
```

### 标准化工作流
提示词确保了无论谁在工作：
-   CI 失败都得到一致的分析。
-   状态更新都遵循团队的格式。
-   后续行动都与流程保持一致。
-   通知都包含所需的信息。

### 学习成果
-   Webhook 集成模式。
-   为保持一致性而进行的提示词工程 (Prompt Engineering)。
-   事件驱动的架构。
-   标准化的团队工作流程。

## 模块 4：Hugging Face Hub 集成

### 我们正在构建什么
与 Hugging Face Hub 集成，用于处理 LLM 和数据集的 PR，为使用语言模型的团队添加专门的工作流程。

### 关键组件

#### 1. Hub 特定工具
```python
# Hugging Face 工作流程的工具：
# - analyze_model_changes：检测 LLM 文件的修改
# - validate_dataset_format：检查训练数据的合规性
# - generate_model_card：创建/更新模型文档
# - suggest_hub_template：为 LLM/数据集提供 PR 模板
```

#### 2. Hub 资源
```python
# Hub 上下文的资源：
# - hub://model-cards/ - LLM 卡片模板和示例
# - hub://dataset-formats/ - 训练数据规范
# - hub://community-standards/ - Hub 社区指南
# - hub://license-info/ - 许可证兼容性检查
```

#### 3. LLM 特定提示
```python
# LLM 工作流程的提示词：
# - "分析模型变更" - 理解 LLM 的更新
# - "生成基准摘要" - 创建评估指标
# - "检查数据集质量" - 验证训练数据
# - "起草模型卡片更新" - 更新文档
```

### Hub 特定工作流

当 PR 修改 LLM 文件时：
1.  **工具**：`analyze_model_changes` 检测模型架构的变更。
2.  **资源**：读取 `hub://model-cards/llm-template.md`。
3.  **提示**："生成基准摘要" 创建评估部分。
4.  **工具**：`generate_model_card` 更新文档。
5.  **资源**：检查 `hub://license-info/` 的兼容性。

### 数据集 PR 处理
对于训练数据的更新：
-   验证格式的一致性。
-   检查数据质量指标。
-   更新数据集卡片。
-   建议适当的审查者。

### 学习成果
-   Hugging Face Hub API 的集成。
-   LLM 特定的 PR 工作流程。
-   模型和数据集的文档。
-   社区标准的合规性。

## 模块 5：Slack 通知

### 我们正在构建什么
结合工具、资源和提示，实现自动化的团队通知，完成整个工作流的自动化。

### 关键组件

#### 1. 通信工具
```python
# 三个用于团队更新的工具：
# - send_slack_message：发布到团队频道
# - get_team_members：识别要通知的人员
# - track_notification_status：监控交付状态
```

#### 2. 团队资源
```python
# 用于团队数据的资源：
# - team://members/ - 开发者档案和偏好设置
# - slack://channels/ - 频道配置
# - notification://templates/ - 消息格式
```

#### 3. 通知提示词
```python
# 用于通信的提示词：
# - "格式化团队更新" - 适当地设置消息样式
# - "选择通信频道" - 选择正确的受众
# - "如果紧急则上报" - 处理紧急问题
```

### 集成示例

当一个关键 PR 的 CI 失败时：
1.  **工具**：`get_team_members` 识别 PR 的作者和审查者。
2.  **资源**：`team://members/{user}/preferences` 检查通知设置。
3.  **提示词**："格式化团队更新" 创建适当的消息。
4.  **工具**：`send_slack_message` 将消息传递到正确的频道。
5.  **资源**：`notification://templates/ci-failure` 确保格式一致。
6.  **提示词**："如果紧急则上报" 确定是否需要额外的警报。

### 智能路由
系统会考虑：
-   团队成员的可用性 (来自日历资源)。
-   通知偏好 (电子邮件 vs. Slack)。
-   消息的紧急程度 (基于 PR 标签)。
-   时区和工作时间。

### 学习成果
-   原语的集成模式。
-   复杂工作流的编排。
-   在自动化与人类需求之间取得平衡。
-   生产就绪的错误处理。

## 完整工作流示例

以下是所有组件如何协同工作以处理一个典型的 PR：

1.  **开发者创建 PR**
    -   GitHub Webhook 触发服务器。
    -   工具：`analyze_file_changes` 检查差异。
    -   资源：读取团队指南和项目上下文。
    -   提示：建议最佳的 PR 模板。

2.  **CI/CD 流水线运行**
    -   Webhook 接收工作流事件。
    -   提示："分析 CI 结果" 处理结果。
    -   资源：检查团队的上报策略。
    -   工具：在 GitHub 中更新 PR 状态。

3.  **Hugging Face Hub 集成**
    -   工具：检测 LLM/数据集的变更。
    -   资源：读取 Hub 的指南。
    -   提示词：生成模型卡片的更新。
    -   工具：根据 Hub 标准进行验证。

4.  **团队通知**
    -   工具：识别相关的团队成员。
    -   资源：读取通知偏好。
    -   提示词：格式化适当的消息。
    -   工具：通过 Slack 频道发送。

5.  **后续行动**
    -   提示词："创建后续任务" 生成下一步行动。
    -   工具：如果需要，创建 GitHub 问题。
    -   资源：链接到相关文档。
    -   所有原语无缝协作。

## 运行解决方案

1.  **启动 MCP 服务器**：`python server.py`

## 故障排查指南

### 常见问题

1.  **Webhook 未接收到事件**
    -   检查 Cloudflare Tunnel 是否正在运行。
    -   验证 GitHub Webhook 的配置。
    -   确认密钥是否匹配。

2.  **工具未出现在 AI 客户端中**
    -   验证工具的模式。
    -   检查服务器的注册情况。
    -   审查 MCP 的连接。

3.  **资源无法访问**
    -   验证文件权限。
    -   检查 URI 的格式。
    -   确认资源的注册情况。

4.  **提示词产生不一致的结果**
    -   审查提示词模板。
    -   检查提供的上下文。
    -   验证输入格式。
