# 第三单元解决方案演练：使用 MCP 构建拉取请求代理

## 概述

本演练将指导你完成第三单元拉取请求代理的完整解决方案——这是一个 MCP 服务器，通过分析代码变更、监控 CI/CD 流水线和自动化团队沟通来帮助开发者创建更好的拉取请求。该解决方案展示了三种 MCP 原语（工具、资源和提示词）在实际工作流程中的协同工作。

## 架构概述

PR 代理由相互连接的模块组成，逐步构建一个完整的自动化系统：

1. **构建 MCP 服务器** - 具有 PR 模板建议工具的基础服务器
2. **智能文件分析** - 使用资源提供项目上下文的增强分析
3. **GitHub Actions 集成** - 使用标准化提示词进行 CI/CD 监控
4. **Hugging Face Hub 集成** - 模型部署和数据集 PR 工作流程
5. **Slack 通知** - 整合所有 MCP 原语的团队沟通

## 模块 1：构建 MCP 服务器

### 我们正在构建什么
一个最小化的 MCP 服务器，使用 MCP 工具分析文件变更并建议适当的 PR 模板。

### 关键组件

#### 1. 服务器初始化（`server.py`）
```python
# 服务器注册三个基本工具：
# - analyze_file_changes：返回关于变更文件的结构化数据
# - get_pr_templates：列出可用模板及其元数据
# - suggest_template：提供智能模板推荐
```

服务器使用 MCP SDK 将这些工具暴露给 Claude Code，使其能够收集信息并就使用哪个 PR 模板做出智能决策。

#### 2. 文件分析工具
`analyze_file_changes` 工具检查 git diff 以识别：
- 文件类型和扩展名
- 变更的文件数量
- 添加/删除的行数
- 常见模式（测试、配置、文档）

这些结构化数据使 Claude 能够理解变更的性质，而无需硬编码决策逻辑。

#### 3. 模板管理
模板以 markdown 文件形式存储在 `templates/` 目录中：
- `bug.md` - 用于错误修复
- `feature.md` - 用于新功能
- `docs.md` - 用于文档更新
- `refactor.md` - 用于代码重构

每个模板都包含 Claude 可以根据分析填充的占位符。

### Claude 如何使用这些工具

1. Claude 调用 `analyze_file_changes` 来了解发生了什么变更
2. 使用 `get_pr_templates` 查看可用选项
3. 使用分析数据调用 `suggest_template`
4. 接收带有推理的推荐
5. 可以根据具体变更自定义模板

### 学习成果
- 理解工具注册和模式
- 让 Claude 使用结构化数据做决策
- 数据收集与决策逻辑的分离

## 模块 2：智能文件分析

### 我们正在构建什么
使用 MCP 资源提供项目上下文和团队指南的增强文件分析。

### 关键组件

#### 1. 资源注册
服务器暴露四种类型的资源：
```python
# 资源提供只读访问：
# - file://templates/ - PR 模板文件
# - file://project-context/ - 编码标准、约定
# - git://recent-changes/ - 提交历史和模式
# - team://guidelines/ - 审查流程和标准
```

#### 2. 项目上下文资源
`project-context/` 目录包含：
- `coding-standards.md` - 特定语言的约定
- `review-guidelines.md` - 审查者关注的内容
- `architecture.md` - 系统设计模式
- `dependencies.md` - 第三方库策略

Claude 可以读取这些内容以了解项目特定的要求。

#### 3. Git 历史分析
`git://recent-changes/` 资源提供：
- 最近的提交消息和模式
- 常见的 PR 标题和描述
- 团队成员贡献模式
- 历史模板使用情况

这有助于 Claude 建议与团队实践一致的模板。

### Claude 如何使用资源

1. 读取 `team://guidelines/review-process.md` 以了解 PR 要求
2. 访问 `file://project-context/coding-standards.md` 获取样式指南
3. 分析 `git://recent-changes/` 以匹配团队模式
4. 将此上下文与文件分析结合以获得更好的建议

### 增强的决策制定
有了资源，Claude 现在可以：
- 建议符合团队约定的模板
- 在 PR 中包含项目特定的要求
- 在描述中引用编码标准
- 与历史团队实践保持一致

### 学习成果
- 资源 URI 设计和模式
- 使项目知识对 AI 可访问
- 上下文感知的决策制定
- 平衡自动化与团队标准

## 模块 3：GitHub Actions 集成

### 我们正在构建什么
使用 webhook 和标准化提示词进行实时 CI/CD 监控，以实现一致的团队沟通。

### 关键组件

#### 1. Webhook 服务器
使用 Cloudflare Tunnel 接收 GitHub Actions 事件：
```python
# Webhook 端点处理：
# - workflow_run 事件
# - check_run 事件
# - pull_request 状态更新
# - 部署通知
```

#### 2. 提示词模板
四个标准化提示词确保一致性：
- **"分析 CI 结果"** - 处理测试失败和构建错误
- **"生成状态摘要"** - 创建人类可读的状态更新
- **"创建后续任务"** - 根据结果建议下一步
- **"起草团队通知"** - 为不同受众格式化更新

#### 3. 事件处理流水线
1. 从 GitHub 接收 webhook
2. 解析事件数据并提取相关信息
3. 根据事件类型使用适当的提示词
4. 生成标准化响应
5. 存储以供团队通知

### Claude 如何使用提示词

提示词使用示例：
```python
# 当测试失败时，Claude 使用"分析 CI 结果"提示词：
prompt_data = {
    "event_type": "workflow_run",
    "status": "failure",
    "failed_jobs": ["unit-tests", "lint"],
    "error_logs": "...",
    "pr_context": {...}
}

# Claude 生成：
# - 根本原因分析
# - 建议的修复
# - 影响评估
# - 下一步行动
```

### 标准化工作流程
提示词确保无论谁在工作：
- CI 失败得到一致的分析
- 状态更新遵循团队格式
- 后续行动与流程保持一致
- 通知包含所需信息

### 学习成果
- Webhook 集成模式
- 一致性的提示词工程
- 事件驱动架构
- 标准化团队工作流程

## 模块 4：Hugging Face Hub 集成

### 我们正在构建什么
与 Hugging Face Hub 集成，用于 LLM 和数据集 PR，为使用语言模型的团队添加专门的工作流程。

### 关键组件

#### 1. Hub 特定工具
```python
# Hugging Face 工作流程的工具：
# - analyze_model_changes：检测 LLM 文件修改
# - validate_dataset_format：检查训练数据合规性
# - generate_model_card：创建/更新模型文档
# - suggest_hub_template：LLM/数据集的 PR 模板
```

#### 2. Hub 资源
```python
# Hub 上下文的资源：
# - hub://model-cards/ - LLM 卡片模板和示例
# - hub://dataset-formats/ - 训练数据规范
# - hub://community-standards/ - Hub 社区指南
# - hub://license-info/ - 许可证兼容性检查
```

#### 3. LLM 特定提示词
```python
# LLM 工作流程的提示词：
# - "分析模型变更" - 理解 LLM 更新
# - "生成基准摘要" - 创建评估指标
# - "检查数据集质量" - 验证训练数据
# - "起草模型卡片更新" - 更新文档
```

### Hub 特定工作流程

当 PR 修改 LLM 文件时：
1. **工具**：`analyze_model_changes` 检测模型架构变更
2. **资源**：读取 `hub://model-cards/llm-template.md`
3. **提示词**："生成基准摘要"创建评估部分
4. **工具**：`generate_model_card` 更新文档
5. **资源**：检查 `hub://license-info/` 的兼容性

### 数据集 PR 处理
对于训练数据更新：
- 验证格式一致性
- 检查数据质量指标
- 更新数据集卡片
- 建议适当的审查者

### 学习成果
- Hugging Face Hub API 集成
- LLM 特定的 PR 工作流程
- 模型和数据集文档
- 社区标准合规性

## 模块 5：Slack 通知

### 我们正在构建什么
结合工具、资源和提示词的自动化团队通知，实现完整的工作流程自动化。

### 关键组件

#### 1. 通信工具
```python
# 三个团队更新工具：
# - send_slack_message：发布到团队频道
# - get_team_members：识别要通知的人
# - track_notification_status：监控交付状态
```

#### 2. 团队资源
```python
# 团队数据的资源：
# - team://members/ - 开发者档案和偏好
# - slack://channels/ - 频道配置
# - notification://templates/ - 消息格式
```

#### 3. 通知提示词
```python
# 通信的提示词：
# - "格式化团队更新" - 适当地设置消息样式
# - "选择通信频道" - 选择正确的受众
# - "如果关键则升级" - 处理紧急问题
```

### 集成示例

当关键 PR 的 CI 失败时：
1. **工具**：`get_team_members` 识别 PR 作者和审查者
2. **资源**：`team://members/{user}/preferences` 检查通知设置
3. **提示词**："格式化团队更新"创建适当的消息
4. **工具**：`send_slack_message` 传递到正确的频道
5. **资源**：`notification://templates/ci-failure` 确保一致的格式
6. **提示词**："如果关键则升级"确定是否需要额外警报

### 智能路由
系统考虑：
- 团队成员可用性（来自日历资源）
- 通知偏好（电子邮件 vs Slack）
- 消息紧急程度（基于 PR 标签）
- 时区和工作时间

### 学习成果
- 原语集成模式
- 复杂工作流程编排
- 平衡自动化与人类需求
- 生产就绪的错误处理

## 完整工作流程示例

以下是所有组件如何协同工作处理典型 PR：

1. **开发者创建 PR**
   - GitHub webhook 触发服务器
   - 工具：`analyze_file_changes` 检查差异
   - 资源：读取团队指南和项目上下文
   - 提示词：建议最佳 PR 模板

2. **CI/CD 流水线运行**
   - Webhook 接收工作流事件
   - 提示词："分析 CI 结果"处理结果
   - 资源：检查团队升级策略
   - 工具：在 GitHub 中更新 PR 状态

3. **Hugging Face Hub 集成**
   - 工具：检测 LLM/数据集变更
   - 资源：读取 Hub 指南
   - 提示词：生成模型卡片更新
   - 工具：根据 Hub 标准验证

4. **团队通知**
   - 工具：识别相关团队成员
   - 资源：读取通知偏好
   - 提示词：格式化适当的消息
   - 工具：通过 Slack 频道发送

5. **后续行动**
   - 提示词："创建后续任务"生成下一步
   - 工具：如需要创建 GitHub 问题
   - 资源：链接到文档
   - 所有原语无缝协作

## 运行解决方案

## 故障排除指南

### 常见问题

1. **Webhook 未接收事件**
   - 检查 Cloudflare Tunnel 是否运行
   - 验证 GitHub webhook 配置
   - 确认密钥匹配

2. **工具未出现在 Claude 中**
   - 验证工具模式
   - 检查服务器注册
   - 审查 MCP 连接

3. **资源不可访问**
   - 验证文件权限
   - 检查 URI 格式
   - 确认资源注册

4. **提示词产生不一致的结果**
   - 审查提示词模板
   - 检查提供的上下文
   - 验证输入格式
