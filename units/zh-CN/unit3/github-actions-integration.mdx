## 核心概念

### MCP 提示词

提示词是可重用的模板，引导 Claude 完成复杂的工作流。与工具（Claude 自动调用）不同，提示词是用户发起的，提供结构化的指导。

示例用例：
- 一致地分析 CI/CD 结果
- 创建标准化的部署摘要
- 系统地排查故障

### Webhook 集成

你的 MCP 服务器将运行两个服务：
1. MCP 服务器（与 Claude 通信）
2. 端口 8080 上的 webhook 服务器（接收 GitHub 事件）

这使 Claude 能够对实时 CI/CD 事件做出反应！

> [!TIP]
> **架构洞察**：为 MCP 通信和 webhook 处理运行独立的服务是一种清晰的关注点分离。webhook 服务器处理 HTTP 复杂性，而你的 MCP 服务器专注于数据分析和 Claude 集成。

## 实现步骤

### 步骤 1：设置并运行 Webhook 服务器

与模块 1 处理现有文件不同，本模块引入了实时事件处理。起始代码包括：
- **你的模块 1 实现** - 所有现有的 PR 分析工具
- **一个完整的 webhook 服务器**（`webhook_server.py`）- 准备接收 GitHub 事件

1. 安装依赖（与模块 1 相同）：
   ```bash
   uv sync
   ```

2. 启动 webhook 服务器（在单独的终端中）：
   ```bash
   python webhook_server.py
   ```

这个服务器将接收 GitHub webhooks 并将它们存储在 `github_events.json` 中。

**webhook 事件存储的工作原理：**
- 每个传入的 GitHub webhook（推送、拉取请求、工作流完成等）都会追加到 JSON 文件中
- 事件与时间戳一起存储，便于查找最近的活动
- 该文件充当一个简单的事件日志，你的 MCP 工具可以读取和分析
- 不需要数据库 - 所有内容都存储在一个简单、可读的 JSON 格式中

### 步骤 2：连接到事件存储

现在你将把你的 MCP 服务器（来自模块 1）连接到 webhook 数据。这比直接处理 HTTP 请求简单得多 - webhook 服务器完成所有繁重的工作并将事件存储在 JSON 文件中。

添加读取 webhook 事件的路径：

```python
# webhook 服务器存储事件的文件
EVENTS_FILE = Path(__file__).parent / "github_events.json"
```

webhook 服务器处理所有 HTTP 细节 - 你只需要读取 JSON 文件！这种关注点分离使你的 MCP 服务器专注于它最擅长的事情。

> [!TIP]
> **开发提示**：使用文件而不是 HTTP 请求使测试变得更容易。你可以手动向 `github_events.json` 添加事件来测试你的工具，而无需设置 webhooks。

### 步骤 3：添加 GitHub Actions 工具

就像在模块 1 中为文件分析创建工具一样，你现在将为 CI/CD 分析创建工具。这些工具将与你现有的 PR 分析工具一起工作，让 Claude 全面了解代码更改和构建状态。

> [!TIP]
> **注意**：起始代码已经包含了模块 1 的输出限制修复，所以你不会遇到令牌限制错误。专注于本模块的新概念！

实现两个新工具：

1. **`get_recent_actions_events`**：
   - 从 `EVENTS_FILE` 读取
   - 返回最近的事件（最多到限制数量）
   - 如果文件不存在则返回空列表

2. **`get_workflow_status`**：
   - 从文件读取所有事件
   - 过滤 workflow_run 事件
   - 按工作流名称分组并显示最新状态

这些工具让 Claude 分析你的 CI/CD 流水线。

### 步骤 4：创建 MCP 提示词

现在你将添加你的第一个 MCP 提示词！与工具（Claude 自动调用）不同，提示词是帮助用户与 Claude 一致交互的模板。把它们想象成引导 Claude 完成复杂工作流的"对话启动器"。

虽然模块 1 专注于数据访问的工具，但本模块引入了工作流指导的提示词。

实现四个展示不同工作流模式的提示词：

1. **`analyze_ci_results`**：全面的 CI/CD 分析
2. **`create_deployment_summary`**：团队友好的更新
3. **`generate_pr_status_report`**：代码 + CI 组合报告
4. **`troubleshoot_workflow_failure`**：系统化调试

每个提示词应该返回一个包含 Claude 要遵循的清晰指令的字符串。

### 步骤 5：使用 Cloudflare Tunnel 测试

现在到了激动人心的部分 - 用真实的 GitHub 事件测试你扩展的 MCP 服务器！你将一起运行多个服务，就像在真实的开发环境中一样。

1. 启动你的 MCP 服务器（与模块 1 相同的命令）：
   ```bash
   uv run server.py
   ```

2. 在另一个终端中，启动 Cloudflare Tunnel：
   ```bash
   cloudflared tunnel --url http://localhost:8080
   ```

3. 使用隧道 URL 配置 GitHub webhook

4. 使用提示词在 Claude Code 中测试

## 常见问题

### Webhook 未接收事件
- 确保 Cloudflare Tunnel 正在运行
- 检查 GitHub webhook 设置（应该显示最近的交付）
- 验证有效负载 URL 包含 `/webhook/github`

### 提示词不工作
- FastMCP 提示词只是返回字符串
- 确保你的函数用 `@mcp.prompt()` 装饰

### Webhook 服务器问题
- 确保 webhook_server.py 在单独的终端中运行
- 检查端口 8080 是否空闲：`lsof -i :8080`
- 事件文件将在接收到第一个事件时自动创建
