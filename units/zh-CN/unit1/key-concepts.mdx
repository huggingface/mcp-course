# 关键概念和术语

在深入了解模型上下文协议之前，理解构成MCP基础的关键概念和术语非常重要。本节将介绍支撑该协议的基本思想，并为整个课程中讨论MCP实现提供通用词汇。

MCP常被描述为"AI应用的USB-C"。正如USB-C为各种外设连接到计算设备提供了标准化的物理和逻辑接口，MCP为AI模型连接外部能力提供了一致的协议。这种标准化使整个生态系统受益：

- **用户**在AI应用中享受更简单、更一致的体验
- **AI应用开发者**可以轻松集成不断增长的工具和数据源生态系统
- **工具和数据提供商**只需创建一个可在多个AI应用中使用的实现
- 更广泛的生态系统受益于增强的互操作性、创新和减少的碎片化

## 集成问题

**M×N集成问题**指的是在没有标准化方法的情况下，将M个不同的AI应用连接到N个不同的外部工具或数据源所面临的挑战。

### 没有MCP（M×N问题）

如果没有像MCP这样的协议，开发者需要创建M×N个自定义集成——为AI应用与外部能力的每个可能配对创建一个。

![没有MCP](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit1/1.png)

每个AI应用都需要单独与每个工具/数据源集成。这是一个非常复杂且昂贵的过程，给开发者带来大量摩擦，并产生高昂的维护成本。

一旦我们有多个模型和多个工具，集成的数量就会变得难以管理，每个都有自己独特的接口。

![多个模型和工具](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit1/1a.png)

### 使用MCP（M+N解决方案）

MCP通过提供标准接口将其转变为M+N问题：每个AI应用实现一次MCP的客户端，每个工具/数据源实现一次服务器端。这大大降低了集成复杂性和维护负担。

![使用MCP](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit1/2.png)

## MCP核心术语

现在我们理解了MCP解决的问题，让我们深入了解构成MCP协议的核心术语和概念。

> [!TIP]
> MCP是一个像HTTP或USB-C一样的标准，是连接AI应用与外部工具和数据源的协议。因此，使用标准术语对于使MCP有效工作至关重要。
>
> 在记录我们的应用程序并与社区沟通时，我们应该使用以下术语。

### 组件

就像HTTP中的客户端-服务器关系一样，MCP有客户端和服务器。

![MCP组件](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit1/3.png)

- **主机（Host）**：面向用户的AI应用，最终用户直接与之交互。例如Anthropic的Claude Desktop、AI增强的IDE如Cursor、推理库如Hugging Face Python SDK，或使用LangChain或smolagents等库构建的自定义应用。主机发起与MCP服务器的连接，并协调用户请求、LLM处理和外部工具之间的整体流程。

- **客户端（Client）**：主机应用内的一个组件，管理与特定MCP服务器的通信。每个客户端与单个服务器保持1:1连接，处理MCP通信的协议级细节，并充当主机逻辑与外部服务器之间的中介。

- **服务器（Server）**：通过MCP协议公开能力（工具、资源、提示词）的外部程序或服务。

> [!WARNING]
> 许多内容将"客户端"和"主机"互换使用。从技术上讲，主机是面向用户的应用程序，而客户端是主机应用程序内管理与特定MCP服务器通信的组件。

### 能力

当然，应用程序的价值是其提供的能力的总和。因此能力是应用程序最重要的部分。MCP可以连接任何软件服务，但有一些常见能力用于许多AI应用。

| 能力 | 描述 | 示例 |
| ---------- | ----------- | ------- |
| **工具（Tools）** | AI模型可以调用的可执行函数，用于执行操作或检索计算数据。通常与应用程序的用例相关。 | 天气应用的工具可能是返回特定位置天气的函数。 |
| **资源（Resources）** | 提供上下文而无需大量计算的只读数据源。 | 研究助手可能有科学论文的资源。 |
| **提示词（Prompts）** | 预定义的模板或工作流，指导用户、AI模型和可用能力之间的交互。 | 摘要提示词。 |
| **采样（Sampling）** | 服务器发起的请求，要求客户端/主机执行LLM交互，实现递归操作，其中LLM可以审查生成的内容并做出进一步决策。 | 写作应用审查自己的输出并决定进一步完善。 |

| 实体 | 名称 | 描述 |
| --- | --- | --- |
| 工具 | 代码解释器 | 可以执行LLM编写的代码的工具。 |

| 采样 | 代码审查 | 允许LLM审查代码并做出进一步决策的采样。 |
