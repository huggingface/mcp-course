# 关键概念与术语

在深入了解模型上下文协议之前，理解其核心概念与术语至关重要。本节将介绍支撑该协议的基本思想，并为后续课程的讨论提供一套统一的词汇。

MCP 常被比作“AI 应用的 USB-C”。正如 USB-C 为各种外设连接计算设备提供了标准化的物理和逻辑接口一样，MCP 为 AI 模型连接外部功能提供了一致的协议。这种标准化让整个生态系统都能从中受益：

-   **用户**可以在 AI 应用中享受到更简单、更一致的体验。
-   **AI 应用开发者**可以轻松集成一个不断壮大的工具和数据源生态系统。
-   **工具和数据提供商**只需创建一次实现，即可在多个 AI 应用中使用。
-   更广泛的生态系统则得益于增强的互操作性、持续的创新和减少的碎片化。

## 集成问题

**M×N 集成问题**指的是，在没有标准化方法的情况下，将 M 个不同的 AI 应用连接到 N 个不同的外部工具或数据源时所面临的挑战。

### 没有 MCP (M×N 问题)

如果没有像 MCP 这样的协议，开发者就需要为 AI 应用与外部功能的每一个可能组合，创建 M×N 个自定义的集成。

![没有 MCP](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit1/1.png)

每个 AI 应用都需要单独与每个工具或数据源进行集成。这是一个极其复杂且昂贵的过程，给开发者带来了巨大的开发阻力，并产生了高昂的维护成本。

一旦我们拥有多个模型和多个工具，且每个都有自己独特的接口，集成的数量就会变得难以管理。

![多个模型和工具](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit1/1a.png)

### 使用 MCP (M+N 解决方案)

MCP 通过提供一个标准接口，将这个问题简化为了一个 M+N 的问题：每个 AI 应用只需实现一次 MCP 的客户端，而每个工具或数据源也只需实现一次 MCP 的服务器端。这极大地降低了集成的复杂性和维护的负担。

![使用 MCP](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit1/2.png)

## MCP 核心术语

现在我们理解了 MCP 所要解决的问题，让我们来深入了解构成 MCP 协议的核心术语和概念。

> [!TIP]
> MCP 就像 HTTP 或 USB-C 一样，是一个标准，是连接 AI 应用与外部工具和数据源的协议。因此，使用标准化的术语对于保证 MCP 的有效性至关重要。
>
> 在为我们的应用编写文档以及与社区交流时，我们都应该使用以下术语。

### 组件

就像 HTTP 中的客户端-服务器关系一样，MCP 也有客户端和服务器。

![MCP 组件](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit1/3.png)

-   **主机 (Host)**：面向用户的 AI 应用，最终用户直接与之交互。例如 Anthropic 的 Claude Desktop、AI 增强的 IDE (如 Cursor)、推理库 (如 Hugging Face Python SDK)，或者使用 LangChain、smol-agents 等库构建的自定义应用。主机负责发起与 MCP 服务器的连接，并协调用户请求、LLM 处理和外部工具使用之间的整体流程。

-   **客户端 (Client)**：主机应用内的一个组件，负责管理与特定 MCP 服务器的通信。每个客户端与单个服务器保持一对一的连接，处理 MCP 通信的协议级细节，并充当主机逻辑与外部服务器之间的中介。

-   **服务器 (Server)**：通过 MCP 协议提供功能 (如工具、资源、提示词) 的外部程序或服务。

> [!WARNING]
> 很多资料会互换使用“客户端”和“主机”这两个术语。从技术上讲，主机是面向用户的应用程序，而客户端是主机应用程序内部，负责管理与特定 MCP 服务器通信的组件。

### 功能 (Capabilities)

当然，一个应用的价值是其提供的所有功能价值的总和。因此，功能是应用中最重要的部分。MCP 可以连接任何软件服务，但有一些常见的功能被许多 AI 应用所使用。

| 功能 | 描述 | 示例 |
| --- | --- | --- |
| **工具 (Tools)** | AI 模型可以调用的可执行函数，用于执行操作或检索计算密集型数据。通常与应用的核心用例相关。 | 天气应用的工具可能是一个返回特定地点天气的函数。 |
| **资源 (Resources)** | 提供上下文信息而无需大量计算的只读数据源。 | 一个研究助手可能需要一个包含科学论文的资源库。 |
| **提示词 (Prompts)** | 预定义的模板或工作流，用于引导用户、AI 模型和可用功能之间的交互。 | 一个用于生成摘要的提示词。 |
| **采样 (Sampling)** | 由服务器发起的请求，要求客户端/主机执行一次 LLM 交互。这可以实现递归操作，即 LLM 可以审查其生成的内容并作出进一步决策。 | 一个写作应用可以审查自己生成的内容，并决定如何进一步完善。 |

在下图中，我们可以看到这些功能集成在一个代码智能体用例中的具体应用。

![集成图表](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit1/8.png)

此应用可能会按以下方式使用其 MCP 实体：

| 实体 | 名称 | 描述 |
| --- | --- | --- |
| 工具 | 代码解释器 | 一个可以执行由 LLM 编写的代码的工具。 |
| 资源 | 文档 | 一个包含应用文档的资源。 |
| 提示词 | 代码风格指南 | 一个指导 LLM 生成符合特定风格代码的提示词。 |
| 采样 | 代码审查 | 允许 LLM 审查生成的代码并作出进一步决策的采样过程。 |
