# 第一单元回顾

## 模型上下文协议 (MCP)

MCP 是一种标准化协议，旨在将 AI 模型与外部工具、数据源和环境相连接。它通过实现互操作性并提供对实时信息的访问，解决了现有 AI 系统的局限性。

## 核心概念

### 客户端-服务器架构
MCP 遵循客户端-服务器模型，其中客户端负责管理用户与服务器之间的通信。这种架构提升了系统的模块化程度，使得在不改动现有宿主的情况下，可以轻松添加新的服务器。

### 组件
#### 宿主 (Host)
宿主是面向用户的 AI 应用程序，作为最终用户的主要交互界面。

##### 客户端 (Client)
客户端是宿主应用程序中的一个组件，负责管理与特定 MCP 服务器的通信。一个客户端与一个服务器保持一一对应的连接，并处理协议层面的细节。

#### 服务器 (Server)
服务器是通过 MCP 协议提供对工具、数据源或服务访问的外部程序或服务。服务器通常是对现有功能的轻量级封装。

### 功能 (Capabilities)

#### 工具 (Tools)
工具是可执行的函数，能够执行具体操作 (例如发送消息、查询 API)。工具通常由模型控制，并且由于其可能产生副作用，执行前需要用户批准。

#### 资源 (Resources)
资源是用于检索上下文的只读数据源，通常不涉及大量计算。资源由应用程序控制，专为数据检索而设计，类似于 REST API 中的 GET 请求。

#### 提示词 (Prompts)
提示词是预定义的模板或工作流，用于引导用户、AI 模型和可用功能之间的交互。提示词由用户控制，为交互设定上下文。

#### 采样 (Sampling)
采样是服务器发起的 LLM 处理请求，它支持由服务器驱动的 Agent 行为，并可能实现递归或多步骤的复杂交互。采样操作通常也需要用户批准。

### 通信协议
MCP 协议使用 JSON-RPC 2.0 作为客户端与服务器之间通信的消息格式。它支持两种主要的传输机制：`stdio` (用于本地通信) 和 `HTTP+SSE` (用于远程通信)。消息类型包括请求、响应和通知。

### 发现机制
MCP 允许客户端通过特定的列表方法 (例如 `tools/list`) 动态发现可用的工具、资源和提示词。这种动态发现机制使得客户端能够适应每个服务器提供的特定功能，而无需硬编码服务器的具体实现。

### MCP SDK
官方为多种编程语言提供了 SDK，用于实现 MCP 客户端和服务器。这些 SDK 封装了协议层面的通信、功能注册和错误处理，从而简化了开发流程。

### Gradio 集成
Gradio 可以帮助您轻松创建 Web 界面，并将功能暴露给 MCP 协议，使其能够同时被人类和 AI 模型访问。这种集成可以用最少的代码，为您封装的功能同时提供人机交互界面和 AI 可调用的工具。
