# MCP 的架构组件

在上一节中，我们讨论了 MCP 的关键概念和术语。现在，让我们深入了解构成 MCP 生态系统的各个架构组件。

## 宿主、客户端与服务器

模型上下文协议 (MCP) 建立在客户端-服务器架构之上，为 AI 模型与外部系统之间提供了一套结构化的通信规范。

![MCP 架构](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit1/4.png)

MCP 架构由三个主要组件构成，每个组件都有明确定义的角色和职责：宿主、客户端和服务器。我们在上一节已对这些组件进行了简要介绍，现在让我们来更深入地了解每个组件及其具体职责。

### 宿主 (Host)

**宿主**是面向用户的 AI 应用程序，终端用户可以直接与之交互。

常见的例子包括：
- AI 聊天应用，如 OpenAI 的 ChatGPT 或 Anthropic 的 Claude Desktop
- AI 增强的集成开发环境 (IDE)，如 Cursor，或与 Continue.dev 等工具的集成
- 使用 LangChain 或 smol-agents 等库构建的自定义 AI 代理和应用

宿主的职责包括：
- 管理用户交互和权限
- 通过 MCP 客户端发起与 MCP 服务器的连接
- 协调用户请求、LLM 处理和外部工具使用之间的整体流程
- 以连贯的格式将结果呈现给用户

在大多数情况下，用户会根据自己的需求和偏好来选择宿主应用。例如，开发者可能会选择 Cursor 以利用其强大的代码编辑功能，而领域专家则可能会使用基于 smol-agents 构建的自定义应用。

### 客户端 (Client)

**客户端**是宿主应用内的一个组件，负责管理与特定 MCP 服务器的通信。其主要特性包括：

- 每个客户端与单个服务器保持一对一的连接
- 处理 MCP 通信的协议级细节
- 充当宿主逻辑与外部服务器之间的中介

### 服务器 (Server)

**服务器**是通过 MCP 协议向 AI 模型提供功能的外部程序或服务。服务器的特点如下：

- 提供对特定外部工具、数据源或服务的访问
- 通常是对现有功能的轻量级封装
- 可以在本地运行 (与宿主在同一台机器上)，也可以远程运行 (通过网络)
- 以标准化的格式提供其功能，以便客户端可以动态发现和使用

## 通信流程

让我们来看看这些组件在典型的工作流程中是如何交互的：

> [!TIP]
> 在下一节中，我们将通过实际示例深入探讨支撑这些组件的通信协议。

1.  **用户交互**：用户与**宿主**应用交互，表达一个意图或查询。

2.  **宿主处理**：**宿主**处理用户的输入，可能会借助 LLM 来理解请求，并判断可能需要哪些外部功能。

3.  **客户端连接**：**宿主**指示其**客户端**组件连接到相应的服务器。

4.  **功能发现**：**客户端**查询**服务器**，以发现其提供的功能 (工具、资源、提示词等)。

5.  **功能调用**：根据用户的需求或 LLM 的判断，**宿主**指示**客户端**调用**服务器**的特定功能。

6.  **服务器执行**：**服务器**执行被请求的功能，并将结果返回给**客户端**。

7.  **结果集成**：**客户端**将结果传递回**宿主**，**宿主**再将这些结果整合到 LLM 的上下文中，或直接呈现给用户。

这种架构的一个关键优势是其模块化的特性。单个**宿主**可以通过不同的**客户端**同时连接到多个**服务器**。新的**服务器**可以轻松地加入到生态系统中，而无需对现有的**宿主**进行任何更改。各项功能也可以方便地在不同的**服务器**之间进行组合。

> [!TIP]
> 正如我们在上一节所讨论的，这种模块化将传统的 M×N 集成问题 (M 个 AI 应用连接到 N 个工具/服务) 转化为一个更易于管理的 M+N 问题，即每个宿主和服务器只需实现一次 MCP 标准即可。

这种架构看似简单，但其强大之处在于通信协议的标准化以及组件之间职责的清晰分离。这种设计催生了一个有凝聚力的生态系统，使得 AI 模型能够无缝地连接到一个不断增长的外部工具和数据源矩阵。

## 总结

这些交互模式遵循了几个关键的设计原则，这些原则也塑造了 MCP 的设计与演进。该协议强调**标准化**，为 AI 连接提供通用协议；同时保持**简洁性**，使核心协议简单但支持高级功能。**安全性**被置于优先地位，敏感操作需要明确的用户批准；而**可发现性**则使得功能的动态发现成为可能。该协议在设计时充分考虑了**可扩展性**，通过版本控制和功能协商来支持其自身的演进，并确保不同实现和环境之间的**互操作性**。
