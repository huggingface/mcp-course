# 理解 MCP 的能力

MCP 服务器通过其通信协议向客户端提供多种能力。这些能力主要分为四大类，每一类都有其独特的特性和适用场景。让我们来深入探索这些构成 MCP 能力基础的核心“原语”。

> [!TIP]
> 在本节中，我们将以独立于具体框架的函数形式来展示示例代码。这样做的目的是为了让您专注于概念本身及其协同工作的方式，而不是陷入某个特定框架的实现细节中。
>
> 在接下来的单元中，我们将展示这些概念如何在 MCP 的特定代码中实现。

## 工具 (Tools)

工具是 AI 模型可以通过 MCP 协议调用的可执行函数或操作。

-   **控制权**：工具通常是**由模型控制的**，这意味着 AI 模型 (LLM) 会根据用户的请求和上下文，自行决定何时调用它们。
-   **安全性**：由于工具能够执行可能产生副作用的操作，因此其执行可能存在风险。所以，它们通常需要用户进行明确的批准。
-   **适用场景**：发送消息、创建工单、查询 API、执行计算等。

**示例**：一个获取指定地点当前天气数据的天气工具：

## 资源 (Resources)

资源提供对数据源的只读访问，允许 AI 模型在不执行复杂逻辑的情况下检索上下文信息。

-   **控制权**：资源是**由应用控制的**，这意味着通常由宿主应用程序来决定何时访问它们。
-   **性质**：它们专为数据检索而设计，计算量极小，类似于 REST API 中的 GET 请求。
-   **安全性**：由于它们是只读的，通常比工具带来的安全风险更低。
-   **适用场景**：访问文件内容、检索数据库记录、读取配置信息等。

**示例**：一个提供文件内容访问的资源：

<hfoptions id="resource-example">
<hfoption id="python">

```python
def read_file(file_path: str) -> str:
    """读取指定路径文件的内容。"""
    with open(file_path, 'r') as f:
        return f.read()
```

</hfoption>
<hfoption id="javascript">

```javascript
function readFile(filePath) {
    // 使用 fs.readFile 读取文件内容
    const fs = require('fs');
    return new Promise((resolve, reject) => {
        fs.readFile(filePath, 'utf8', (err, data) => {
            if (err) {
                reject(err);
                return;
            }
            resolve(data);
        });
    });
}
```

## 提示词 (Prompts)

提示词是预定义的模板或工作流，用于引导用户、AI 模型和服务器能力之间的交互。

-   **控制权**：提示词是**由用户控制的**，通常在宿主应用程序的 UI 中作为选项呈现给用户。
-   **目的**：它们为交互构建了一个结构，以优化对可用工具和资源的利用。
-   **选择**：用户通常在 AI 模型开始处理之前选择一个提示词，从而为整个交互设定好上下文。
-   **适用场景**：常见的自动化工作流、专业任务模板、引导式交互等。

**示例**：一个用于生成代码审查的提示词模板：

## 采样 (Sampling)

采样允许服务器向客户端 (特指宿主应用) 请求执行一次 LLM 交互。

-   **控制权**：采样是**由服务器发起的**，但需要客户端/宿主的协助才能完成。
-   **目的**：它使得由服务器驱动的 Agent 行为，以及潜在的递归或多步骤交互成为可能。
-   **安全性**：与工具类似，采样操作通常也需要用户批准。
-   **适用场景**：复杂的多步骤任务、自主 Agent 工作流、交互式流程等。

**示例**：服务器可能会请求客户端分析其已经处理过的数据：

<hfoptions id="sampling-example">
<hfoption id="python">

```python
def request_sampling(messages, system_prompt=None, include_context="none"):
    """向客户端请求一次 LLM 采样。"""
    # 在实际实现中，这将向客户端发送一个请求
    return {
        "role": "assistant",
        "content": "这是对所提供数据的分析结果..."
    }
```

采样流程遵循以下步骤：
1.  服务器向客户端发送一个 `sampling/createMessage` 请求。
2.  客户端审查该请求，并可以对其进行修改。
3.  客户端从 LLM 进行采样。
4.  客户端审查采样完成的结果。
5.  客户端将最终结果返回给服务器。

> [!TIP]
> 这种“人在环路中”(human-in-the-loop) 的设计确保了用户始终能控制 LLM 看到和生成的内容。在实现采样时，提供清晰、结构良好的提示词并包含相关上下文非常重要。

## 各项能力如何协同工作

让我们来看看这些能力是如何协同工作以实现复杂交互的。在下表中，我们概述了各项能力、其控制方、控制方向以及其他一些细节。

| 能力 | 控制方 | 方向 | 副作用 | 需要批准 | 典型适用场景 |
| --- | --- | --- | --- | --- | --- |
| 工具 | 模型 (LLM) | 客户端 → 服务器 | 是 (可能) | 是 | 执行操作、API 调用、数据操作 |
| 资源 | 应用程序 | 客户端 → 服务器 | 否 (只读) | 通常否 | 数据检索、上下文收集 |
| 提示词 | 用户 | 服务器 → 客户端 | 否 | 否 (由用户选择) | 引导式工作流、专业模板 |
| 采样 | 服务器 | 服务器 → 客户端 → 服务器 | 间接 | 是 | 多步骤任务、 Agent 行为 |

这些能力被设计成以互补的方式协同工作：

1.  用户可能会选择一个**提示词**来启动一个专业的工作流。
2.  该提示词可能会包含从**资源**中获取的上下文信息。
3.  在处理过程中，AI 模型可能会调用**工具**来执行特定的操作。
4.  对于复杂的操作，服务器可能会使用**采样**来请求额外的 LLM 处理。

这些核心能力之间的区别为 MCP 的交互提供了清晰的结构，使得 AI 模型能够在保持适当控制边界的同时，访问信息、执行操作并参与复杂的工作流。

## 发现机制

MCP 的一个关键特性是能力的动态发现。当客户端连接到服务器时，它可以通过特定的列表方法来查询可用的工具、资源和提示词：

-   `tools/list`：发现可用的工具
-   `resources/list`：发现可用的资源
-   `prompts/list`：发现可用的提示词

这种动态发现机制允许客户端适应每个服务器提供的特定能力，而无需硬编码服务器的具体实现。

## 总结

理解这些核心原语对于有效使用 MCP 至关重要。通过提供具有明确控制边界的不同类型的能力，MCP 在保持适当安全和控制机制的同时，实现了 AI 模型与外部系统之间的强大交互。
