# 实现 Webhook 监听器

Webhook 监听器是我们“拉取请求代理”的入口。当 Hugging Face Hub 上的讨论被创建或更新时，它会接收实时事件，从而触发我们基于 MCP 的智能标签工作流。在本节中，我们将使用 FastAPI 来实现一个功能强大的 Webhook 处理器。

## 理解 Webhook 集成

遵循 [Hugging Face Webhooks 指南](https://raw.githubusercontent.com/huggingface/hub-docs/refs/heads/main/docs/hub/webhooks-guide-discussion-bot.md)，我们的 Webhook 监听器会验证传入的请求并实时处理讨论事件。

![Webhook 创建流程](https://huggingface.co/datasets/huggingface/documentation-images/resolve/main/hub/webhooks-guides/001-discussion-bot/webhook-creation.png)

### Webhook 事件流程

理解 Webhook 的工作流程，对于构建一个可靠的监听器至关重要：

1. **用户操作**：有人在模型仓库的讨论区中创建了一条评论。
2. **Hub 事件**：Hugging Face Hub 生成一个 Webhook 事件。
3. **Webhook 传递**：Hub 向我们的应用端点发送一个 POST 请求。
4. **身份验证**：我们验证 Webhook 密钥以确保请求的安全性。
5. **处理**：从评论内容中提取标签建议。
6. **执行操作**：调用 MCP 工具为新标签创建拉取请求。

> [!TIP]
> Webhook 是一种“推送”通知机制——Hugging Face Hub 会主动将事件发送到您的应用，而不是由您去轮询变更。这使得实时响应讨论和评论成为可能。

## 构建 FastAPI Webhook 应用

让我们从基础开始，一步步构建我们的 Webhook 监听器。

### 1. 应用设置

首先，我们来设置一个基础的 FastAPI 应用，并导入所有必要的模块。

```python
import os
import json
from datetime import datetime
from typing import List, Dict, Any, Optional

from fastapi import FastAPI, Request, BackgroundTasks
from fastapi.middleware.cors import CORSMiddleware
from pydantic import BaseModel
```

这些导入为我们构建一个健壮的 Webhook 处理器提供了所有必需的工具。`FastAPI` 提供了 Web 框架，`BackgroundTasks` 支持异步处理，而类型提示则有助于进行数据验证。

现在，我们来配置应用：

```python
# 配置
WEBHOOK_SECRET = os.getenv("WEBHOOK_SECRET")
HF_TOKEN = os.getenv("HF_TOKEN")

# 用于存储已处理操作的简单列表
tag_operations_store: List[Dict[str, Any]] = []

app = FastAPI(title="HF Tagging Bot")
app.add_middleware(CORSMiddleware, allow_origins=["*"])
```

此配置完成了以下设置：
- **Webhook 密钥**：用于验证传入的 Webhook 请求。
- **HF 令牌**：用于与 Hub API 进行身份验证。
- **操作记录存储**：一个内存列表，用于监控已处理的操作。
- **CORS 中间件**：允许来自 Web 界面的跨域请求。

> [!TIP]
> `tag_operations_store` 列表用于追踪最近的 Webhook 处理操作。这在调试和监控时非常有用。但在生产环境中，您可能需要考虑使用数据库或限制此列表的大小，以防止内存溢出。

### 2. Webhook 数据模型

基于 [Hugging Face Webhook 文档](https://raw.githubusercontent.com/huggingface/hub-docs/refs/heads/main/docs/hub/webhooks-guide-discussion-bot.md)，我们需要理解 webhook 数据结构：

```python
class WebhookEvent(BaseModel):
    event: Dict[str, str]          # 包含操作和范围信息
    comment: Dict[str, Any]        # 评论内容和元数据
    discussion: Dict[str, Any]     # 讨论信息
    repo: Dict[str, str]           # 仓库详情
```

这个 Pydantic 模型帮助我们更好地理解和验证 Webhook 的结构。

我们关心的关键字段是：
- `event.action`：通常为新评论的 "create"
- `event.scope`：通常为评论事件的 "discussion.comment"
- `comment.content`：实际的评论文本
- `repo.name`：发表评论的仓库

### 3. 核心 Webhook 处理器

现在，我们来实现主要的 Webhook 处理器——这是魔法发生的地方。

```python
@app.post("/webhook")
async def webhook_handler(request: Request, background_tasks: BackgroundTasks):
    """
    处理来自 Hugging Face Hub 的传入 webhooks
    遵循以下模式：https://raw.githubusercontent.com/huggingface/hub-docs/refs/heads/main/docs/hub/webhooks-guide-discussion-bot.md
    """
    print("🔔 收到 Webhook！")
    
    # 步骤 1：验证 webhook 密钥（安全性）
    webhook_secret = request.headers.get("X-Webhook-Secret")
    if webhook_secret != WEBHOOK_SECRET:
        print("❌ 无效的 Webhook 密钥")
        return {"error": "incorrect secret"}, 400
```

第一步是安全验证。我们检查 `X-Webhook-Secret` 请求头是否与我们配置的密钥匹配，以确保 Webhook 请求的合法性。

> [!TIP]
> **始终验证 Webhook 密钥！**
> 没有这个检查，任何人都可以向您的应用发送伪造的 Webhook 请求。这个共享密钥扮演着在 Hugging Face 和您的应用之间进行身份验证的“密码”角色。

接下来，我们解析并验证 Webhook 数据：

```python
    # 步骤 2：解析 Webhook 数据
    try:
        webhook_data = await request.json()
        print(f"📥 Webhook 数据：{json.dumps(webhook_data, indent=2)}")
    except Exception as e:
        print(f"❌ 解析 Webhook 数据时出错：{str(e)}")
        return {"error": "invalid JSON"}, 400
    
    # 步骤 3：验证事件结构
    event = webhook_data.get("event", {})
    if not event:
        print("❌ Webhook 中缺少事件数据")
        return {"error": "missing event data"}, 400
```

这个解析步骤能够优雅地处理潜在的 JSON 格式错误，并验证我们是否收到了预期的事件结构。

现在是事件过滤逻辑：

```python
    # 步骤 4：检查这是否是讨论评论创建
    # 遵循 webhook 指南模式：
    if (
        event.get("action") == "created" and 
        event.get("scope") == "discussion.comment"
    ):
        print("✅ 这是一个有效的“讨论评论创建”事件")
        
        # 在后台处理，以便快速向 Hub 返回响应
        background_tasks.add_task(process_webhook_comment, webhook_data)
        
        return {
            "status": "accepted",
            "message": "评论处理任务已开始",
            "timestamp": datetime.now().isoformat()
        }
    else:
        print(f"ℹ️ 忽略事件：action={event.get('action')}, scope={event.get('scope')}")
        return {
            "status": "ignored",
            "reason": "非“讨论评论创建”事件"
        }
```

这个过滤器确保我们只处理我们关心的事件——新的讨论评论。我们忽略其他类型的事件，如仓库创建、模型上传等。

我们使用 FastAPI 的 `background_tasks.add_task()` 来异步处理 Webhook。这使我们能够迅速返回响应，而实际的标签处理则在后台进行。

> [!TIP]
> Webhook 端点应在 10 秒内响应，否则发送平台可能会认为请求超时。使用后台任务可以确保快速响应，同时允许复杂的处理逻辑异步执行。

### 4. 评论处理逻辑

现在，我们来实现核心的评论处理函数，它将执行实际的标签提取和 MCP 工具调用：

```python
async def process_webhook_comment(webhook_data: Dict[str, Any]):
    """
    处理 Webhook 评论，以检测和添加标签。
    该函数与我们的 MCP 客户端集成，以进行 Hub 交互。
    """
    print("🏷️ 开始 process_webhook_comment...")
    
    try:
        # 提取评论和仓库信息
        comment_content = webhook_data["comment"]["content"]
        discussion_title = webhook_data["discussion"]["title"]
        repo_name = webhook_data["repo"]["name"]
        discussion_num = webhook_data["discussion"]["num"]
        comment_author = webhook_data["comment"]["author"].get("id", "unknown")
        
        print(f"📝 来自 {comment_author} 的评论：{comment_content}")
        print(f"📰 讨论：{discussion_title}")
        print(f"📦 仓库：{repo_name}")
```

这部分代码从 Webhook 数据中提取所有相关信息。我们同时获取评论内容和讨论标题，因为标签可能出现在任何一个地方。

接下来，我们提取并处理标签：


我们合并来自两个来源的标签，并创建一个操作记录用于监控。这个记录可以帮助我们追踪每个 Webhook 处理操作的进度。

> [!TIP]
> 存储操作记录对于调试和监控至关重要。当出现问题时，您可以查看最近的操作记录来了解发生了什么以及原因。

现在是与 MCP Agent 集成的部分：

```python
        if not all_tags:
            operation["status"] = "no_tags"
            operation["message"] = "No recognizable tags found"
            print("❌ 没有找到要处理的标签")

        for tag in all_tags:

                prompt = f"""
                分析仓库 '{repo_name}' 并判断是否应添加标签 '{tag}'。
                
                首先，使用 get_current_tags 检查现有标签。
                如果 '{tag}' 尚不存在且为一个有效标签，则使用 add_new_tag 添加它。
                
                仓库：{repo_name}
                待处理标签：{tag}
                
                请提供您所执行操作的清晰摘要。
                """
                
                response = await agent.run(prompt)
                print(f"🤖 Agent 对 '{tag}' 的响应：{response}")
                
                # 解析响应并存储结果
                tag_result = {
                    "tag": tag,
                    "response": response,
                    "timestamp": datetime.now().isoformat()
                }
                operation["results"].append(tag_result)
                
            except Exception as e:
                error_msg = f"❌ 处理标签 '{tag}' 时出错：{str(e)}"
                print(error_msg)
                operation["results"].append({
                    "tag": tag,
                    "error": str(e),
                    "timestamp": datetime.now().isoformat()
                })
        
        operation["status"] = "completed"
        print(f"✅ 完成处理 {len(all_tags)} 个标签")
```

这部分是核心业务逻辑：
1. **验证**：确保我们有需要处理的标签和一个可用的 Agent。
2. **处理**：为每个标签创建一个自然语言提示词，并交由 Agent 执行。
3. **记录**：存储所有操作结果，以便监控和调试。
4. **错误处理**：优雅地处理单个标签处理过程中可能出现的错误。

Agent 的提示词经过精心设计，旨在引导 AI 准确地执行所需步骤：首先检查现有标签，然后在适当时添加新标签。

## 在 Hugging Face Hub 上配置 Webhook

现在我们的 Webhook 监听器已经准备就绪，让我们在 Hugging Face Hub 上进行配置，将我们的应用与真实的仓库事件连接起来。

### 1. 在设置中创建 Webhook

遵循 [Webhook 设置指南](https://huggingface.co/docs/hub/webhooks-guide-discussion-bot)：

![Webhook 设置](https://huggingface.co/datasets/huggingface/documentation-images/resolve/main/hub/webhooks-guides/001-discussion-bot/webhook-creation.png)

导航到您的 [Hugging Face 设置](https://huggingface.co/settings/webhooks) 并进行如下配置：

1. **目标仓库**：指定要监控的仓库
2. **Webhook URL**：你部署的应用程序端点（例如，`https://your-space.hf.space/webhook`）
3. **密钥**：使用与你的 `WEBHOOK_SECRET` 环境变量相同的密钥
4. **事件**：订阅"社区（PR 和讨论）"事件

> [!TIP]
> 在为大量仓库配置 Webhook 之前，最好先从一两个测试仓库开始。这能让您在扩大应用范围之前，验证一切是否正常工作。

### 2. Space URL 配置

对于部署在 Hugging Face Spaces 上的应用，您需要获取其“直接 URL”：

![直接 URL](https://huggingface.co/datasets/huggingface/documentation-images/resolve/main/hub/webhooks-guides/001-discussion-bot/direct-url.png)

过程如下：
1. 在您的 Space 设置中点击“嵌入此 Space”。
2. 复制“直接 URL”。
3. 在 URL 末尾附加 `/webhook` 来创建您的 Webhook 端点。
4. 使用这个完整的 URL 来更新您的 Webhook 配置。
