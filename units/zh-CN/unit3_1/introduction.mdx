## 我们的拉取请求代理项目

我们将构建一个标签代理，它由四个主要组件组成：MCP 服务器、webhook 监听器、代理逻辑和部署基础设施。该代理将能够根据讨论和评论为模型仓库添加标签。这应该可以为模型作者节省时间，让他们收到可以直接使用的 PR，而不必手动为其仓库添加标签。

![PR 代理架构](https://huggingface.co/datasets/mcp-course/images/resolve/main/unit3/architecture.png)

在上图中，我们有一个可以读取和更新模型标签的 MCP 服务器。我们有一个可以接收来自 Hugging Face Hub 的 webhook 的 webhook 监听器。我们有一个可以分析讨论和评论并创建 PR 以更新模型标签的代理。我们有一个可以将 MCP 服务器部署到 Hugging Face Spaces 的部署基础设施。

### MCP 服务器 (`mcp_server.py`)

我们应用程序的核心 - 一个 FastMCP 服务器，提供以下工具：
- 从模型仓库读取当前标签
- 通过拉取请求向 Hub 添加新标签
- 错误处理和验证

这是你将实现 MCP 服务器并为此项目完成大部分工作的地方。Gradio 应用和 FastAPI 应用将用于测试 MCP 服务器和 webhook 监听器，它们已经可以使用。

### Webhook 集成

遵循 [Hugging Face Webhooks 指南](https://huggingface.co/docs/hub/webhooks-guide-discussion-bot)，我们的代理：
- 监听讨论评论事件
- 验证 webhook 签名以确保安全
- 处理提及和标签建议
- 自动创建拉取请求

### 代理功能

代理分析讨论内容以：
- 提取显式标签提及（`tag: pytorch`、`#transformers`）
- 从自然语言中识别隐式标签
- 根据已知的 ML/AI 类别验证标签
- 生成适当的拉取请求描述

### 部署和生产

- 容器化部署到 Hugging Face Spaces
- 密钥的环境变量管理
- webhook 响应的后台任务处理
- 用于测试和监控的 Gradio 界面

## Webhook 集成概述

我们的 PR 代理利用了 Hugging Face 讨论机器人使用的相同 webhook 基础设施。以下是 webhook 如何实现实时响应：

![Webhook 流程](https://huggingface.co/datasets/huggingface/documentation-images/resolve/main/hub/webhooks-guides/001-discussion-bot/webhook-creation.png)

webhook 流程如下：
1. **事件触发**：用户在模型仓库讨论中创建评论
2. **Webhook 传递**：Hugging Face 向我们的端点发送 POST 请求
3. **身份验证**：我们验证 webhook 密钥以确保安全
4. **处理**：我们的代理分析评论以获取标签建议
5. **操作**：如果找到相关标签，我们创建拉取请求
6. **响应**：webhook 立即返回，而 PR 创建在后台进行
